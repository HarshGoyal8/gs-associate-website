# GS Associate Website — Changelog

All notable changes to this project will be documented in this file.

This project follows a simplified version of the **Keep a Changelog** specification and uses **Semantic Versioning (SemVer)** where appropriate.

---

# Versioning Strategy

The project follows the format:

```text
MAJOR.MINOR.PATCH
```

Example:

```text
0.1.0
```

Where:

* **MAJOR** → Breaking architectural changes
* **MINOR** → New features
* **PATCH** → Bug fixes, improvements, documentation updates

# Change Log Rules

Every completed feature must update this document.

Each release should include:

* Added
* Changed
* Fixed
* Removed (if applicable)
* Documentation (if applicable)

Do not skip releases.

Do not rewrite previous entries.

Always append new changes.

---

# Release Workflow

Every release follows this sequence:

1. Complete sprint tasks.
2. Verify functionality.
3. Update TASKS.md.
4. Update CHANGELOG.md.
5. Commit changes.
6. Push to GitHub.
7. Deploy to Vercel.
8. Verify production deployment.
9. Tag the release.

# Guiding Principle

The changelog is the historical record of the project.

Every meaningful change should be documented so that anyone can understand how the project has evolved, why decisions were made, and what was delivered in each release.

A well-maintained changelog makes the project easier to maintain, debug, and extend over time.
